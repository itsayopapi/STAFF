Getting Started with GitHub

Learn the fundamentals of GitHub integration and set up your first repository with these step-by-step guides

Create a Repository

Initialize a new repository locally or on GitHub to start version controlling your project.

# Initialize new repo
git init

# Connect to remote
git remote add origin https://github.com/user/repo.git

Branching Strategy

Understand how to use branches effectively to manage different versions of your code.

# Create new branch
git branch feature-branch

# Switch to branch
git checkout feature-branch

# Push branch to remote
git push -u origin feature-branch

Pull Requests

Learn how to propose changes and collaborate through pull requests.

# Create pull request from CLI
gh pr create --title "New feature" \
--body "Description of changes" \
--base main --head feature-branch

GitHub CLI Tool

The GitHub CLI brings GitHub to your terminal, allowing you to:

  • Run your entire GitHub workflow from terminal
  • Create and review pull requests
  • Manage issues and notifications
  • Interact with GitHub Actions
# Install GitHub CLI
brew install gh

Git Flow

The Git Flow workflow defines a strict branching model designed around the project release. This workflow doesn't require any other tools besides Git.

When to Use:
  • Projects with scheduled version releases
  • Teams needing to maintain multiple versions in production
  • Projects with complex release cycles

GitHub Flow

GitHub Flow is a lightweight, branch-based workflow that supports teams and projects with regular deployments.

1. Create a Branch

Create a descriptively named branch from your main branch for your work.

git checkout -b my-feature main
2. Add Commits

Make changes and commit them to your branch with clear, descriptive messages.

git commit -m "Add new authentication feature"
3. Open a Pull Request

Create a pull request to propose your changes and start a discussion.

4. Discuss and Review

Collaborate with your team through code review comments and discussions.

5. Deploy for Testing

Deploy your branch to a testing environment to verify your changes.

6. Merge to Main

Once approved, merge your changes into the main branch.

git checkout main
git merge my-feature

Continuous Integration & Deployment

Automate your build, test, and deployment processes with GitHub Actions

GitHub Actions

GitHub Actions makes it easy to automate all your software workflows, now with world-class CI/CD. Build, test, and deploy your code right from GitHub.

Key Features:
  • Automate workflows based on GitHub events
  • Run jobs in parallel on multiple OS environments
  • Use community-created actions or build your own
  • Seamless integration with GitHub packages
  • Matrix builds for testing multiple versions
  • Caching for faster builds

Sample Workflows

name: Node.js CI on: push: branches: [ main ] pull_request: branches: [ main ] jobs: build: runs-on: ubuntu-latest strategy: matrix: node-version: [14.x, 16.x] steps: - uses: actions/checkout@v2 - name: Use Node.js ${{ matrix.node-version }} uses: actions/setup-node@v2 with: node-version: ${{ matrix.node-version }} - run: npm install - run: npm test

name: Docker Build on: push: branches: [ main ] jobs: build: runs-on: ubuntu-latest steps: - uses: actions/checkout@v2 - name: Build Docker image run: docker build -t my-app . - name: Log in to Docker Hub uses: docker/login-action@v1 with: username: ${{ secrets.DOCKER_HUB_USERNAME }} password: ${{ secrets.DOCKER_HUB_TOKEN }} - name: Push to Docker Hub run: | docker tag my-app ${{ secrets.DOCKER_HUB_USERNAME }}/my-app:latest docker push ${{ secrets.DOCKER_HUB_USERNAME }}/my-app:latest

name: Deploy to AWS on: push: branches: [ main ] jobs: deploy: runs-on: ubuntu-latest steps: - uses: actions/checkout@v2 - uses: actions/setup-node@v2 with: node-version: '16' - run: npm install - run: npm run build - uses: aws-actions/configure-aws-credentials@v1 with: aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }} aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }} aws-region: us-east-1 - run: aws s3 sync ./build s3://my-bucket --delete

GitHub Best Practices

Follow these guidelines to get the most out of GitHub

Repository Management

README Files

Every repository should have a comprehensive README.md file that explains:

  • Project purpose and goals
  • Installation instructions
  • Usage examples
  • Contribution guidelines
  • License information
.gitignore

Use a .gitignore file to prevent sensitive or unnecessary files from being tracked:

# Dependency directories
node_modules/

# Environment files
.env

# IDE files
.vscode/
.idea/
Branch Protection

Enable branch protection rules for critical branches (main, develop) to:

  • Require pull request reviews
  • Require status checks to pass
  • Require linear history
  • Include administrators

Collaboration Practices

Commit Messages

Follow the conventional commits specification for clear, standardized messages:

feat: add new authentication module

fix: resolve login page crash

docs: update API documentation

chore: update dependencies

Use imperative mood ("add" not "added" or "adds") and limit to 50 characters.

Pull Requests

Effective PR practices for better collaboration:

  • Keep PRs small and focused
  • Use descriptive titles
  • Include screenshots for UI changes
  • Link to related issues
  • Request specific reviewers
  • Use templates for consistency
Issue Tracking

Manage issues effectively with:

  • Clear, descriptive titles
  • Labels for categorization
  • Milestones for tracking
  • Assignees for accountability
  • Templates for bug reports and feature requests

GitHub Security Best Practices

Access Control
  • Use teams for permission management
  • Follow principle of least privilege
  • Enable 2FA for all accounts
  • Regularly review third-party app access
Code Security
  • Enable Dependabot for dependency updates
  • Use CodeQL for static analysis
  • Scan for secrets in commits
  • Require signed commits for critical repos
